# 实验记录

## 阶段一：高层优化（现已回退）
- **使用 `torch.compile` 和 `StaticCache`**:
  - `StaticLayer.update` monkey patch 导致吞吐暴降（54 tok/s -> 16 tok/s）。
  - `DynamicLayer.update` 分块预分配：性能略降。

## 阶段二：纯手工 Kernel 优化（保留并巩固）
- **融合 QKV + RoPE + RMSNorm**:
  - 成功将 4 个独立算子合一，避免中间显存读写，单步提速稳定。
- **融合 MLP Gate + Up Projection**:
  - 将两个 Linear 合并为一，使用一次 GEMM 完成，吞吐稳定在 **42.21 tok/s**。

## 阶段三：调度/内存/算子 三维手搓引擎（最新成果）
严格遵循不量化、不改结构、不使用投机解码与外接框架的规则。我们手写了推理引擎 `manual_kernel_plus_fastpath`，包含：
1. **调度层 (Scheduling)**:
   - 彻底剥离 `transformers` 原生的 `model.generate` 函数。
   - 手写了基于 **CUDA Graph** 的单步 Token 解码循环，彻底消除了每次生成新词时巨大的 Python / CPU 调度下发延迟 (Launch Overhead)。
2. **内存层 (Memory)**:
   - 彻底废弃动态 `torch.cat` 分配的 KV Cache。
   - 在模型初始化时预分配一块全局连续的 **StaticCache** (固定容量 4096)。
   - 在 CUDA Graph 的 Replay 循环中，直接通过 GPU 指针原子更新 KV，实现零内存碎片、零 CPU 分配开销，且**图结构可跨样本全局复用**，分摊了 Graph 捕获成本。
3. **算子层 (Operator)**:
   - 在静态连续显存的支持下，我们将 Decode 阶段的 `scaled_dot_product_attention` 彻底替换为了针对连续静态缓存优化的 `flash_attn_with_kvcache` 接口。
   - 配合前期的 Triton 融合算子，将算子延迟压榨到了极限。

### 性能结果：
在 **10 个真实样本** 上，在不强制延长生成长度（保留真实 EOS 提早停止）的严苛官方口径下：
- **TTFT (首字延迟)**: ~116.41 ms (保持极速)
- **Decode 吞吐**: 从 42.21 tok/s 飙升至 **129.75 tokens/sec**
- **综合性能提升**: **~3.07 倍**

这一套“手搓 CUDA Graph 引擎 + 全局连续内存池 + FlashAttention”方案，是纯手工在底层优化大模型的终极解法之一。
