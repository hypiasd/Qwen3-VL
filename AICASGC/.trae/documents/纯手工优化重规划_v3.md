# Qwen3-VL 纯手工优化重规划 (Phase 3) - 深度底层突破

## 1. 现状回顾 (Current Status Review)
- **调度层 (Scheduling)**: 已实现 Dual Static Graph (CUDA Graph) 消除 Python 下发开销，并结合了 Draft-Free 的 Prompt Lookup Decoding (PLD) 技术。
- **内存层 (Memory)**: 已实现基于 `transformers.cache_utils.StaticCache` 的预分配静态显存池，并利用 Triton 实现原地 (inplace) `update`，消除了 `torch.cat` 的显存碎片和分配开销。
- **算子层 (Operator)**:
  - 文本端：接入了 `flash_attn_with_kvcache`；融合了 QKV 投影 (`_aicas_qkv_weight`)；实现了 Fused RMSNorm + RoPE (`triton_fused_rmsnorm_rope`)；实现了 MLP Gate/Up 投影融合；实现了极致的 Epilogue Fusion (`fused_residual_rmsnorm`) 以突破单 batch 显存带宽墙。
  - 视觉端：实现了视觉 QKV + RoPE 融合 (`triton_vision_qkv_rope_transpose`)；实现了视觉 LayerNorm 与 GELUTanh 的 Triton 融合。
- **当前吞吐**: 经 `test_throughput_128.py` 测试，单 batch 吞吐量达到 **~261.8 tok/s**。

## 2. 深入瓶颈分析 (Deep Bottleneck Analysis)
针对“有人是我的两倍吞吐量”的目标，且在**不利用数据集重复特征取巧**的前提下，进一步提升吞吐必须从最底层的算法和算子极限榨取性能。目前的瓶颈主要在于：
1. **单调的 PLD 匹配算法限制了接受率**: 当前的 Prompt Lookup Decoding 采用固定的 3-gram 匹配，并在固定窗口 (K=15) 下查找。若上下文较长或匹配模式复杂，定长的 N-gram 极易截断更长的潜在匹配序列，导致每次投机生成的有效 Token 数（Acceptance Rate）未达到上限。
2. **首字延迟 (TTFT) 的 Python 下发开销**: Decode 阶段已完全 CUDA Graph 化，但 Prefill 阶段（尤其是 Vision Encoder 处理上千个 Patch 时）由于序列长度动态变化，仍存在大量的 PyTorch Python 侧算子下发开销（CPU Launch Overhead），这部分开销在单 Batch 下严重拖慢了首字时间，进而拉低了总体 `throughput`。
3. **部分算子的访存冗余**: 虽然我们已经做了 Epilogue Fusion，但在 QKV 投影后进行 RoPE 和写入 KV Cache 的过程中，数据仍然要在 SRAM 和 HBM 之间多走一趟。

## 3. 下一阶段优化规划 (Phase 3 Optimization Plan)

### 3.1. 调度层：多级动态 N-gram PLD (Dynamic N-gram Prompt Lookup)
在不引入外部草稿模型的前提下，进一步强化投机解码：
- **当前痛点**: 固定的 `ngram_size=3` 会漏掉许多较短但有效的 2-gram 匹配，或者打断本来能匹配 5-gram 的超长前缀。
- **优化方案**:
  - **动态长度匹配**: 在每次 Decode 循环中，同时匹配长度为 2, 3, 4, 5 的后缀（可以利用高效的 Tensor `unfold` 广播机制并行计算）。
  - **最长优先策略**: 选择匹配长度最长且命中候选数最多的序列进行 Speculative Graph 投机，最大化每次投机的期望收益（Expected Accepted Tokens）。
  - **期望收益**: 通过提高单次 Draft-Free Speculative 的命中率，将单 Batch 的 Decode 阶段极限吞吐推向 350+ tok/s。

### 3.2. 引擎层：Prefill 阶段的 CUDA Graph 分桶 (Bucketed Prefill Graph)
- **当前痛点**: Prefill 的序列长度（图像 Patch + 文本 Token）是动态的，导致原生 `generate` 必须走 Eager 模式，耗时较高。
- **优化方案**:
  - 针对常见的 Vision + Text 序列长度，预先定义几个常用的 Bucket（如 1024, 2048, 4096）。
  - 在首次运行到对应 Bucket 时，通过 Padding 将序列对齐到 Bucket 长度，并**捕获并缓存该长度的 Prefill CUDA Graph**。
  - **期望收益**: 将 Prefill 阶段的 CPU Launch Overhead 彻底消除，进一步压缩 TTFT（首字时间），从而大幅提升平均吞吐公式 (`token_count / total_time`) 的数值。

### 3.3. 算子层：深度前缀融合与 RoPE 优化 (Advanced Prologue Fusion)
- **Fused RoPE inside Attention (Eliminating Memory Bound)**:
  - 探索重写或升级 Triton Attention Kernel，使得 Q 和 K 的旋转位置编码 (RoPE) 直接在 FlashAttention 的 SRAM 中 On-The-Fly 计算，彻底省去中间将 RoPE 结果写回 HBM 的步骤。
- **MLP Down-Proj Prologue Fusion**:
  - 当前已经融合了 Gate 和 Up 投影。探索利用 Triton 将 `SiLU` 激活函数与后置的 `down_proj` 全连接层进行 Prologue Fusion，进一步逼近硬件的显存带宽理论极限。

## 4. 落地实施路线 (Execution Roadmap)
1. **[最高优] 增强多级 PLD 匹配算法**: 修改 `evaluation_wrapper.py` 中的 `fast_generate` 匹配逻辑，实现多长度 N-gram 的并行搜索，直接提升 Decode 的极限吞吐。
2. **[中优] Prefill Graph 分桶拦截**: 在包装器中引入 Prefill 阶段的 CUDA Graph 捕获与 Padding 机制，大幅优化首字时间 (TTFT)。
3. **[探索] 极致的 RoPE + Attention 融合**: 尝试进一步压榨内存带宽，合并最后剩余的散碎算子。
