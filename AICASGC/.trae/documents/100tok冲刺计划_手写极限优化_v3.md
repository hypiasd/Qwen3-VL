# 100tok冲刺计划（先测瓶颈，再做手写优化）

## Summary

* 目标：在不改 `benchmark.py`、不做量化的前提下，把吞吐推进到 **100 tok/s**。

* 新要求：按你的反馈，**先做瓶颈测试，再落地优化**，避免盲改。

* 边界：仅修改 `evaluation_wrapper.py`，其余代码保持不变。

* 验收：以官方 benchmark 全量流程为准。

## Current State Analysis

* 当前 `evaluation_wrapper.py` 已启用 compile/SDPA/static cache/TF32/fastpath 组合。

* 既有实验记录显示吞吐约 89\~90 tok/s，距离 100 tok/s 仍有缺口。

* 历史热点信息指向 decode 路径（GEMM + cat/copy），但结论跨批次存在波动，需先做同口径复测。

* benchmark 约束固定为 `model.generate()` 路径和 128 token 吞吐统计，不允许修改 benchmark。

## Assumptions & Decisions

* 决策1：仅改 `evaluation_wrapper.py`（用户已确认）。

* 决策2：不使用量化（用户已确认“不能量化，该权重”）。

* 决策3：目标按 **100 tok/s 硬目标**推进（用户已确认）。

* 决策4：最终以官方 benchmark 全量结果为唯一验收口径（用户已确认）。

* 决策5：执行顺序调整为“瓶颈测试 → 计划收敛 → 代码优化”，先证据后改动。

* 假设1：评测环境与仓库说明一致（A800 + torch2.8/cu128），本地趋势可用于策略筛选。

## Proposed Changes

### 阶段A：先做瓶颈测试（不改代码）

#### A1) 官方口径基线复测

* 运行官方 benchmark 全量，采集 TTFT/吞吐/准确率三项基线。

* 目标：锁定当前真实起点与波动区间，避免历史样本口径差异干扰。

#### A2) Decode 微观瓶颈复测

* 使用现有 `decode_hotspot_profile.py` 对当前配置做多样本热点采样。

* 输出：prefill/decode 延迟分布、top CUDA ops、cat/copy 与 GEMM 占比。

* 目标：确认是否仍是“GEMM + 内存搬运”主导，且量化可优化空间。

#### A3) 可疑项对照测试

* 对 compile、static cache、fastpath 的关键开关做最小对照组。

* 目标：把“收益项/噪声项/负收益项”分清，作为后续改造顺序依据。

### 阶段B：基于测试结论实施手写优化（仅 evaluation\_wrapper.py）

#### B1) 生成路径重构（最小注入）

* 将 generate 包装简化为最小必要参数注入，去除可疑复制/分支。

#### B2) Decode 热路径定向优化

* 按阶段A热点结果，优先处理 decode 主损耗（形状稳定化、复制减负、调度稳定化）。

#### B3) 编译与缓存策略重排

* 不盲目保留整模 compile；按对照结果选择“整模/局部/关闭”最优组合。

* 围绕 static cache 做稳定化处理，仅保留真实正收益配置。

#### B4) 冲刺档与回退档固化

* 在 wrapper 内固定少量策略档位，默认启用冲刺档，保留安全回退档。

## Verification Steps

* 阶段A验收（先测）：

  * 官方 benchmark 全量基线完成且结果可复现。

  * decode 热点报告完成并给出可量化瓶颈排序。

* 阶段B验收（再改）：

  * 官方 benchmark 全量：`avg_tokens_per_second >= 100`。

  * 真实性校验：无伪造 token、无 benchmark 特判。

  * 稳定性校验：重复流程无崩溃，指标波动可接受。

## Execution Order

1. 先执行阶段A：官方全量基线 + decode 热点 + 关键开关对照。
2. 基于阶段A结果锁定优先级，再实施阶段B的 wrapper 手写优化。
3. 用官方全量做达标验证与回归验证，未达 100 则继续按瓶颈迭代。
