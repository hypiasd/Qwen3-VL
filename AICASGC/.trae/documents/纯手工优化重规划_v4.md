# Qwen3-VL 纯手工优化重规划 (Phase 4) - Prefill 与 L2 Cache 深度榨取

## 1. 现状回顾 (Current Status Review)

* **前期优化**: 已实现基于 CUDA Graph 的双图多级 N-gram (5, 4, 3, 2) 投机解码、全局 StaticCache、Attention / Vision QKV+RoPE / Vision LayerNorm+GELUTanh / Text MLP Gate+Up / Text Epilogue (Residual+RMSNorm) 的全方位 Triton 极核融合。

* **当前吞吐**: 在不违规（如缓存视觉特征）的前提下，基于 `benchmark.py` 测试 10 样本的真实平均吞吐量已达到 **206.41 tok/s**，TTFT 降低至约 **104.46 ms**。

## 2. 深入瓶颈分析 (Deep Bottleneck Analysis)

针对“继续分析并生成优化方案”的目标，目前单 Batch 生成的 Decode 阶段已经极其接近物理显存带宽的理论上限（Memory Bandwidth Wall），进一步优化的重心将转移至：

1. **Prefill 阶段的视觉与文本计算效率 (Vision & Text Prefill Efficiency)**:

   * 当前在 `evaluation_wrapper.py` 的初始化中，`AutoModelForImageTextToText.from_pretrained` 并没有显式指定 `attn_implementation="flash_attention_2"`。这意味着如果环境未默认启用，预填充阶段（尤其是 Vision Encoder 处理上千个 Patch 时）可能会回退到低效的 `sdpa` 或 `eager` 模式。

   * **验证结论**: 经脚本 `test_fa_vision.py` 验证，若不显式指定 `flash_attention_2`，视觉编码器默认采用 `sdpa`。
2. **Decode 阶段的 L2 Cache 驻留 (L2 Cache Residency)**:

   * 在逐 Token 生成时，KV Cache 的频繁读写会消耗大量带宽。如果能利用 Triton 的 `eviction_policy` 将常用的 Prompt KV Cache 或模型权重驻留在 GPU L2 Cache 中，可微弱降低 HBM 访存压力。

## 3. 下一阶段优化规划 (Phase 4 Optimization Plan)

### 3.1. 强制启用全局 FlashAttention-2 (Global FlashAttention-2)

* **优化方案**:

  * 在 `evaluation_wrapper.py` 的 `load_kwargs` 中，强制加入 `"attn_implementation": "flash_attention_2"`。

  * **预期收益**: 这将直接让预填充 (Prefill) 阶段的文本与视觉双端的 Attention 计算全部走最高效的 FlashAttention 路径，进一步压榨首字时间 (TTFT)。

### 3.2. L2 Cache 访问策略优化 (L2 Cache Eviction Policy Tuning)

* **优化方案**:

  * 审视 `triton_kernels.py` 中的自定义算子（如 `_silu_mul_kernel`, `_fused_residual_rmsnorm_kernel` 等）。

  * 对于一次性读取且后续不再使用的中间激活，可以考虑设置 `eviction_policy="evict_first"`。

  * 对于会被频繁读取的权重，确保它们能更好地利用 L2 Cache。

### 3.3. 进一步清理与验证 (Cleanup and Verification)

* 检查是否存在其他可被优化的细碎 Python 开销（如 `torch.cat` 等）。

* 使用 `benchmark.py` 复测吞吐与 TTFT，验证是否能突破 210 tok/s。

## 4. 落地实施路线 (Execution Roadmap)

1. **修改模型加载参数**: 在 `evaluation_wrapper.py` 的 `from_pretrained` 调用中增加 `"attn_implementation": "flash_attention_2"`。
2. **优化 Triton 算子访存策略**: 在 `triton_kernels.py` 中为合适的 `tl.load` 增加 `eviction_policy` 提示。
3. **性能复测**: 运行官方 benchmark，记录优化收益。
