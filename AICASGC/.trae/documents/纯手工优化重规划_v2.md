# Qwen3-VL 纯手工极致优化重规划 (v2)

根据官方的严苛约束：**不能量化、不改结构、不使用投机解码、禁用成熟框架（如 vLLM/SGLang 等），仅允许从「调度」、「内存」、「算子」三个维度进行手写优化**。

基于我们前期的实验和最新的架构级测试，现制定以下纯手工极致优化方案：

## 一、 调度维度优化 (Scheduling) —— 手写 CUDA Graph 极简推理引擎
**瓶颈分析**：
原生 `transformers` 的 `generate` 函数包含海量的 Python 控制流逻辑（如各类 LogitsProcessor、StoppingCriteria、BeamSearch 分支等）。在 Decode 阶段，由于每次只处理 1 个 token，GPU 计算时间极短，**CPU 下发 kernel 的开销 (Launch Overhead) 成为绝对瓶颈**。

**手写方案**：
1. **剥离框架控制流**：绕过 `model.generate`，手写一个专为 Greedy Search 定制的极简 Auto-Regressive 循环。
2. **CUDA Graph 静态捕获**：
   - 在 Prefill 结束后，使用虚拟的 `next_token` 和 `cache_position` 进行 3 次 Warmup 预热。
   - 使用 `torch.cuda.CUDAGraph()` 手工录制单步 Decode 的执行图。
   - 在生成循环中，仅需调用 `g.replay()` 并在显存内原地 (in-place) 更新输入游标，**实现零 CPU 调度开销**。
3. **预期收益**：经初步原型验证，此方案可将 Decode 速度从约 42 tok/s 跃升至 **150~170 tok/s**。

## 二、 内存维度优化 (Memory) —— 连续静态预分配 (Static KV Cache)
**瓶颈分析**：
原生模型默认使用 `DynamicCache`，每生成一个 token 就会调用 `torch.cat` 将新的 KV 拼接到旧的 KV 上。这在长序列下会导致频繁的显存申请、释放与大规模内存搬运，且无法被 CUDA Graph 录制。

**手写方案**：
1. **全生命周期预分配**：在 Prefill 阶段，根据 `seq_len + max_new_tokens` 一次性向 GPU 申请一块连续且固定大小的显存作为 KV Cache 池。
2. **游标寻址 (In-place Update)**：
   - 维护一个全局的 `cache_position` 游标（驻留在 GPU 端）。
   - 在每次生成新 token 时，将新的 K 和 V 投影直接按索引写入预分配的显存切片中，彻底消除 `torch.cat` 搬运开销。
3. **配合 Graph 锁定地址**：静态内存与 CUDA Graph 天然契合，Graph 录制时会固化显存地址，使得每次 replay 都能无缝读写正确的物理地址。

## 三、 算子维度优化 (Operator) —— 极致 Kernel-Level 融合
**瓶颈分析**：
我们虽然已经去除了大部分高层开销，但 PyTorch 默认算子的细碎读写（Memory Bound）仍会消耗宝贵的带宽。

**手写方案**：
1. **保留并巩固已验证的融合算子**：
   - **Fused RMSNorm + RoPE Triton Kernel**：将 Attention 前的层归一化、Q/K 切分与旋转位置编码融合在同一个 Kernel 的寄存器内完成。
   - **MLP 双线性融合**：将 `gate_proj` 和 `up_proj` 的矩阵乘法进行拼版合并，减少一次 GEMM 启动。
2. **优化 SDPA 内存格式**：
   - 确保传入 `scaled_dot_product_attention` 的 tensor 是内存连续的 (contiguous)。
   - 手动剔除在单 token Decode 阶段不必要的 Attention Mask 掩码计算（因为在 Decode 时不需要做 Causal Mask 遮罩，全量计算即可），进一步精简算子计算量。
