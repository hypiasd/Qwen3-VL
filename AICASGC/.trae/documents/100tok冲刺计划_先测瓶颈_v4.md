# 100tok冲刺计划（先测瓶颈，再改代码）

## Summary
- 目标：在不改 `benchmark.py`、不量化、不改权重的前提下，把吞吐优化到 **100 tok/s**。
- 约束：仅修改 `/workspace/Qwen3-VL/AICASGC/evaluation_wrapper.py`。
- 方法：先完成同口径瓶颈测试，基于证据决定优化顺序，再实施手写优化。
- 验收：以官方 benchmark 全量流程结果为唯一准绳。

## Current State Analysis
- 当前优化实现位于 [evaluation_wrapper.py](file:///workspace/Qwen3-VL/AICASGC/evaluation_wrapper.py)，已包含 compile/SDPA/static cache/TF32/fastpath。
- 历史记录显示当前吞吐约 89~90 tok/s，尚未达到 100 tok/s。[实验记录.md](file:///workspace/Qwen3-VL/AICASGC/实验记录.md#L582-L601)
- 已有热点分析指出 decode 路径的 GEMM 与内存搬运（`cat/copy_`）是主要开销。[实验记录.md](file:///workspace/Qwen3-VL/AICASGC/实验记录.md#L462-L510)
- benchmark 规则明确：不能改 benchmark、核心改动在 wrapper、吞吐口径固定。[README_CN.md](file:///workspace/Qwen3-VL/AICASGC/README_CN.md#L202-L213)

## Assumptions & Decisions
- 决策1：先执行瓶颈测试，再制定并落地优化（用户明确要求）。
- 决策2：实现层面只动 `evaluation_wrapper.py`，不新增外部推理框架依赖。
- 决策3：保持真实生成路径，禁止任何 benchmark 特判或伪造 token 的捷径。
- 决策4：100 tok/s 作为硬目标；若单轮未达标，按瓶颈排序迭代。
- 假设1：本地测试趋势可用于指导评测机优化方向，但最终以官方评测机结果为准。

## Proposed Changes

### 阶段A：瓶颈测试（不改代码）

#### A1) 官方全量基线复测
- 目的：锁定当前真实基线（TTFT/吞吐/准确率）及单次波动。
- 操作：
  - 使用官方 benchmark 全量命令运行当前代码。
  - 记录结果文件中的 `avg_ttft_ms`、`avg_tokens_per_second`、`accuracy`、`score`。
- 产出：基线结果 JSON + 基线摘要（作为后续对照基准）。

#### A2) decode 微观热点复测
- 目的：确认当前主瓶颈是否仍集中在 decode 的 GEMM 与内存搬运。
- 操作：
  - 使用现有 [decode_hotspot_profile.py](file:///workspace/Qwen3-VL/AICASGC/decode_hotspot_profile.py) 对当前配置做多样本采样。
  - 汇总 `prefill_ms`、`decode_avg_ms`、`top_cuda_ops`。
- 产出：热点报告（按 self CUDA time 排序的 Top op 列表）。

#### A3) 关键开关最小对照
- 目的：确认 compile/static cache/fastpath 的真实边际贡献和副作用。
- 操作：
  - 只做最小数量对照组，避免实验矩阵过大：默认组、去 compile 组、去 fastpath 组。
  - 对每组重复官方 benchmark（同口径）并记录吞吐变化。
- 产出：对照表（每项开关的增益/负收益与稳定性结论）。

### 阶段B：基于测试结果的手写优化（仅改 evaluation_wrapper.py）

#### B1) 生成包装最小化
- 做法：精简 generate patch，仅保留确定正收益项；删除引入额外 copy 或行为不稳定的逻辑。
- 目标：减少 decode 每步额外开销与不必要参数处理。

#### B2) decode 热路径定向优化
- 做法：按 A2 排名优先处理热点，优先削减 `cat/copy_` 相关路径与形状抖动导致的额外成本。
- 目标：把 decode 平均时延压到达到 100 tok/s 所需区间。

#### B3) compile 与 static cache 重排
- 做法：以 A3 结果为准重排策略（保留/局部化/关闭 compile；保留并稳定 static cache）。
- 目标：在吞吐收益和稳定性之间取得最佳点，避免无效优化叠加。

#### B4) 固化提交配置
- 做法：保留单一默认“冲刺配置”，仅在 wrapper 内留最小回退机制，不暴露复杂外部控制面。
- 目标：提交版本可复现、可解释、可维护。

## Verification Steps
- 验证1（阶段A完成判据）：
  - 完成官方全量基线结果；
  - 完成 decode 热点报告；
  - 完成关键开关最小对照表。
- 验证2（阶段B完成判据）：
  - 官方 benchmark 全量 `avg_tokens_per_second >= 100`；
  - 准确率与总分无明显异常下滑；
  - 连续复测稳定，无新崩溃或非法访存。
- 验证3（真实性判据）：
  - 输出 token 长度与生成过程符合官方路径；
  - 无样本特判、无 benchmark 定向捷径。

## Execution Order
1. 跑 A1 官方全量基线并固化记录。
2. 跑 A2 decode 热点并出瓶颈排序。
3. 跑 A3 最小对照，明确“保留项/剔除项”。
4. 按 B1→B2→B3 顺序修改 `evaluation_wrapper.py`。
5. 跑 B4 官方全量验收，不达标则按热点结论继续迭代。
