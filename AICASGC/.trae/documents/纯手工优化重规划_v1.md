# 纯手工优化重规划 v1

## Summary
- 目标：在 **不使用外部框架、不使用投机解码、只保留纯手工实现优化** 的前提下，重新规划 `Qwen3-VL/AICASGC` 的性能优化路线，且以 **吞吐优先** 为第一目标。
- 允许范围：允许 `monkey patch`、手写 `forward`/`generate fastpath`、手写 Triton/custom kernel、手写 cache/shape 管理；不把 `torch.compile`、`StaticCache`、`SDPA`、`TF32` 这类开关类优化作为主方案。
- 规划粒度：做到 **文件级实施计划 + 实验矩阵**，让执行阶段可以直接按顺序实现和 A/B。

## Current State Analysis

### 1. 当前仓库里“纯手工实现”的实际落点
- 主文件：
  - `evaluation_wrapper.py`
  - `triton_kernels.py`
  - `实验记录.md`
- 已落地的纯手工实现：
  - 视觉位置编码缓存与 Triton 插值：`evaluation_wrapper.py`
  - Vision QKV + RoPE 融合：`evaluation_wrapper.py` + `triton_kernels.py`
  - Vision LayerNorm / GELU Triton 替换：`evaluation_wrapper.py` + `triton_kernels.py`
  - 语言模型 Attention 前 RMSNorm + RoPE 融合：`evaluation_wrapper.py` + `triton_kernels.py`
  - 全量 RMSNorm Triton 替换：`evaluation_wrapper.py` + `triton_kernels.py`
  - Decoder residual inplace add：`evaluation_wrapper.py`
  - `generate fastpath` 包装：`evaluation_wrapper.py`

### 2. 当前仓库里的“开关类优化”
- `torch.compile`：`evaluation_wrapper.py`
- `SDPA`：`evaluation_wrapper.py`
- `generation_config.cache_implementation = "static"`：`evaluation_wrapper.py`
- `TF32`：`evaluation_wrapper.py`

### 3. 现有实验里可确认的有效性
- 历史最高有效成绩来自“手工实现 + 开关类混合方案”：
  - 约 `TTFT 114.31 ms`
  - 约 `Throughput 90.50 tok/s`
  - 见 `实验记录.md`
- 但若剥离开关类优化，历史证据显示：
  - `generate fastpath` 只有小幅正收益
  - 大部分早期 monkey patch 单项收益接近噪声级
  - 真正显著有效的“纯手工项”主要是 **最新一轮 Triton 算子级融合**
- 另外，最近一次尝试“手写 greedy decode loop 替代原生 generate”时，出现了三类问题：
  - 与 `torch.compile`/CUDA graph 交互时直接崩溃
  - 切到原始 `forward` 后吞吐下降到约 `40 tok/s`
  - 质量和显存占用在长生成路径下都出现退化

### 4. 当前瓶颈判断
- 既然这次规划明确排除 `compile/static cache/SDPA/TF32` 作为主优化手段，新的主战场不应再是“开关叠加”，而应聚焦：
  1. **decode 路径的手工瘦身**
  2. **语言模型侧的内存搬运减少**
  3. **手写 cache 更新路径**
  4. **视觉前缀只做样本内复用，不做跨样本缓存**
- 当前最值得继续深挖的方向不是继续加更多 patch，而是把已有手工优化收敛为两条主线：
  - **算子级重写**
  - **生成路径重写**

## Assumptions & Decisions
- 决策 1：本轮计划中，`torch.compile`、`StaticCache`、`SDPA`、`TF32` 均视为 **非主方案**，后续实施时默认关闭或作为对照组，而不是正式优化主体。
- 决策 2：允许保留和扩展 `monkey patch`，包括：
  - 替换模块 `forward`
  - 替换 `self_attn.forward`
  - 替换 `generate` 外层壳
- 决策 3：吞吐优先，允许 TTFT 有限退化，但不能接受：
  - 空输出
  - 明显质量崩坏
  - 长回答 OOM
- 决策 4：这次计划必须包含 **停手条件**，避免继续在负收益方向上消耗时间。

## Proposed Changes

### A. 收敛优化边界：拆分“纯手工版”配置层

#### 文件
- `evaluation_wrapper.py`

#### 变更
- 把初始化优化路径拆成两套显式配置：
  - `manual_only_profile`
  - `switch_based_profile`（仅作为对照，不作为目标版本）
- `manual_only_profile` 只允许启用：
  - 视觉位置编码缓存
  - Vision QKV+RoPE 融合
  - Vision LayerNorm/GELU 替换
  - 语言模型 RMSNorm+RoPE 融合
  - 全量 RMSNorm 替换
  - residual inplace add
  - 手写 generate fastpath
- `manual_only_profile` 默认禁用：
  - `_enable_torch_compile()`
  - `_enable_sdpa_attention()`
  - `_enable_tensor_float32()`
  - `generation_config.cache_implementation = "static"`

#### 原因
- 先把“纯手工优化”的边界从代码结构上固定下来，否则后续实验无法判断收益归因。

---

### B. 第一主线：重写 generate fastpath，但坚持“渐进式最小替换”

#### 文件
- `evaluation_wrapper.py`

#### 变更
- 不再一次性完全替换 `generate` 全流程，而是分三层递进：

##### B1. 轻量 fastpath（先保正确）
- 保留原生 `generate` 主循环
- 只手动做：
  - 清理无效采样参数
  - `pixel_values.contiguous()`
  - 固定 greedy 路径
  - 减少不必要的 kwargs 传递

##### B2. 半手工 fastpath（只替换 pre/post，不替换 decode loop）
- 目标是把：
  - 输入预处理
  - 输出切片
  - 常见 kwargs 规范化
  从外层手工接管
- 但 decode loop 仍交给原生 `generate`

##### B3. 手工 decode loop（最终候选）
- 仅当 B1/B2 无法继续提升吞吐时，再尝试：
  - 手工 greedy loop
  - 只支持 benchmark 真实用到的：
    - `batch_size=1`
    - `do_sample=False`
    - `use_cache=True`
    - `max_new_tokens in {1, 128, 1024}`
- 但这条线必须满足两个前提才继续：
  - 长生成不 OOM
  - 输出质量不出现明显退化

#### 原因
- 上一次直接手搓完整 loop 已证明风险很高。应从“尽量少替换”开始，逐层收集收益，而不是一次性重写整个生成栈。

---

### C. 第二主线：把“未接入的手工 Triton 内核”变成可 A/B 的候选项

#### 文件
- `triton_kernels.py`
- `evaluation_wrapper.py`

#### 当前未接入但已存在的内核
- `fused_layernorm_linear`
- `triton_static_cache_update`
- `triton_elementwise_mul`

#### 变更
- 不直接全接，而是按风险分级：

##### C1. 低风险候选
- `fused_layernorm_linear`
  - 优先尝试接在语言模型中最热点、且形状稳定的线性前置层
  - 如果无法稳定替换全路径，则先做单点 A/B

##### C2. 中风险候选
- `triton_elementwise_mul`
  - 仅在 profile 里明确确认该类 elementwise 在热点前列时才接

##### C3. 高风险候选
- `triton_static_cache_update`
  - 因本轮主方案明确不依赖 `StaticCache`，这一项默认不进主线
  - 只保留为“如果最终允许少量 cache 辅助机制”时的备用试验

#### 原因
- 仓库已经有手工内核资产，不利用会浪费；但它们不是都适合直接接入主路径，需要分层试验。

---

### D. 第三主线：语言模型侧优先，视觉侧止损

#### 文件
- `evaluation_wrapper.py`
- `triton_kernels.py`

#### 变更
- 纯手工优化优先投到 **语言模型 decode 路径**
- 视觉分支只保留已验证正收益或低风险的项：
  - 保留：视觉位置编码缓存
  - 条件保留：Vision QKV+RoPE、Vision LN/GELU
- 如果后续 A/B 发现视觉侧 patch 在手工 profile 下吞吐无正收益，则优先删除视觉侧 patch，减少复杂度与 graph 交互风险

#### 原因
- benchmark 的吞吐核心来自 decode，不来自单次视觉 prefill
- 视觉侧 patch 即使有效，也更容易只改善 TTFT 或只带来噪声级收益

---

### E. 加入可判定的 profile 结构，避免“越改越乱”

#### 文件
- `evaluation_wrapper.py`

#### 变更
- 把优化分成明确的 profile 组合：

##### E1. `manual_baseline`
- 仅保留：
  - 原生 HF generate
  - 不启用 compile/static cache/SDPA/TF32

##### E2. `manual_kernel_only`
- 在 `manual_baseline` 基础上只启用 Triton 算子 patch

##### E3. `manual_kernel_plus_fastpath`
- 在 `manual_kernel_only` 基础上启用轻量 `generate fastpath`

##### E4. `manual_decode_experimental`
- 在 `manual_kernel_plus_fastpath` 基础上尝试半手工 / 全手工 decode 路径

#### 原因
- 本轮目标是“重新规划”，不是盲改。profile 化能保证每步实验都有归因。

## Experiment Matrix

### Phase 1：建立“纯手工基线”
- 目标：得到不含开关类优化的真实基线
- 配置：
  - 关闭 `compile`
  - 关闭 `SDPA`
  - 关闭 `TF32`
  - 关闭 `static cache`
  - 关闭所有 Triton patch
  - 保留原生 `generate`
- 记录：
  - TTFT
  - Throughput
  - 输出非空率
  - 长回答是否 OOM

### Phase 2：只开算子级手工优化
- 配置：`manual_kernel_only`
- 子实验：
  1. 只开语言模型侧 Triton patch
  2. 只开视觉侧 Triton patch
  3. 语言 + 视觉都开
  4. residual inplace 单独开
- 目标：
  - 找到纯手工 patch 的正收益子集
- 停手条件：
  - 单项吞吐提升 < 1%
  - 或质量波动明显

### Phase 3：只测生成外壳手工优化
- 配置：`manual_baseline + generate_fastpath`
- 目标：
  - 测试外层 wrapper 在无开关辅助时是否仍有稳定小收益

### Phase 4：手工 patch + fastpath 组合
- 配置：`manual_kernel_plus_fastpath`
- 目标：
  - 找“纯手工可提交版”
- 成功标准：
  - 吞吐超过 `manual_baseline`
  - 输出质量正常
  - 无 OOM

### Phase 5：实验性手工 decode
- 配置：`manual_decode_experimental`
- 只在满足以下条件时才继续：
  - Phase 4 已证实 patch 组合稳定
  - 仍有显著性能缺口
- 子实验：
  1. 半手工 generate（不改 decode loop）
  2. 只替换 prefill 到 first token
  3. 完整 greedy loop
- 停手条件：
  - 吞吐低于 `manual_kernel_plus_fastpath`
  - 输出出现重复崩坏
  - 长生成 OOM
  - 任何需要重新引入 `compile/static cache` 才能成立

## File-by-File Implementation Steps

### `evaluation_wrapper.py`
- 增加 profile 概念与清晰的启停边界
- 拆分手工优化与开关类优化
- 改造 `__init__` 中优化顺序，使 profile 可切换
- 保留并整理现有 monkey patch 路径
- 渐进式重写 `generate_fastpath`

### `triton_kernels.py`
- 为已接入内核补充“适用阶段”说明（语言 decode / 视觉 prefill）
- 将未接入内核标注为候选项，后续按实验矩阵接入

### `实验记录.md`
- 新增“纯手工优化重规划”章节
- 按 `manual_baseline / manual_kernel_only / manual_kernel_plus_fastpath / manual_decode_experimental` 四个 profile 记录实验

## Verification Steps
- 基础正确性：
  - 小样本 benchmark（2 样本）验证无报错
  - 输出 JSON 中 `prediction` 非空
- 性能验证：
  - 10 样本 benchmark 对比四个 profile
  - 至少记录：
    - `avg_ttft_ms`
    - `avg_throughput_tokens_per_sec`
- 稳定性验证：
  - `max_new_tokens=1024` 不出现明显重复退化
  - 不出现 OOM
  - 不出现空输出/立即 EOS

## Success Criteria
- 输出一版“纯手工可提交候选配置”，满足：
  - 不依赖 `torch.compile/static cache/SDPA/TF32`
  - 输出质量正常
  - 相比纯手工基线有明确正收益
- 如果所有纯手工路径都无法稳定超过基线，则明确收敛结论：
  - **保留纯手工 patch 的最小正收益子集**
  - 不再继续推进高风险手工 decode 重写
